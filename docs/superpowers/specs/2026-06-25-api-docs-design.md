# API Documentation (OpenAPI + Redoc) — Design

**Date:** 2026-06-25
**Status:** Approved (design); implementation plan pending

## Summary

Document the wallet-api HTTP endpoints by generating an OpenAPI spec from
[swaggo/swag](https://github.com/swaggo/swag) annotations on the existing
handlers, and rendering it with [Redoc](https://github.com/Redocly/redoc) as a
clean, read-only reference. The rendered docs are available in two places: served
by the app at `/docs`, and as a committed standalone page for GitHub Pages /
offline viewing.

## Goals

- A polished, static API reference (Redoc), not an interactive Swagger UI.
- Spec generated from code annotations so response schemas stay in sync with the
  Go types (`wallet.TokenPortfolio`, `wallet.TransactionPage`, etc.).
- Served by the running app at `/docs` (+ `/openapi.json`) and also published as a
  committed static page.
- A CI guard so the generated spec cannot silently drift from the handlers.

## Non-goals

- No interactive "try it out" console (that's Swagger UI; we chose Redoc).
- No hand-written OpenAPI spec (annotations are the single source of truth).
- No change to endpoint behavior, routing semantics, or the response payloads.

## Background: current API

Two read-only endpoints, served by `net/http.ServeMux` (Go 1.22 method+pattern
routing) in `internal/api/router.go`, handled in `internal/api/handlers.go`:

- `GET /v1/addresses/{address}/tokens` → `*wallet.TokenPortfolio`
- `GET /v1/addresses/{address}/transactions?limit=&pageKey=` → `*wallet.TransactionPage`

`{address}` must match `^0x[0-9a-fA-F]{40}$`. `limit` defaults to 25 and is
clamped to 100. Error responses (from `writeServiceError`/`writeError`) use the
body `{"error": "<message>"}` with these statuses:

| Condition | Status | Message |
|---|---|---|
| invalid address | 400 | `invalid address` |
| `wallet.ErrUpstream` | 502 | `upstream provider error` |
| `wallet.ErrStore` | 503 | `storage unavailable` |
| anything else | 500 | `internal error` |

## Tooling decision

- **swag v1** (emits OpenAPI **2.0**). Redoc renders Swagger 2.0 natively, so no
  conversion is needed. swag v2 (native OpenAPI 3.1) is a possible future
  upgrade, noted but not adopted now to keep the toolchain stable.
- **Redoc** for rendering (clean static reference). Loaded via CDN `<script>` in
  both the served page and the static page. Vendoring `redoc.standalone.js` for
  fully-offline rendering is a noted future option, not done now.

## Components

### 1. Annotations

**General API info** — above `main()` in `cmd/server/main.go`:

```go
// @title        wallet-api
// @version      1.0.0
// @description  Read-only, non-custodial EVM wallet API (token portfolio + transaction history), allowlist-filtered via LI.FI.
// @BasePath     /v1
```

**Per-endpoint** — on the two handlers in `internal/api/handlers.go`:

- `getTokens`:
  - `@Summary`/`@Description` for the token portfolio
  - `@Tags addresses`
  - `@Param address path string true "0x-prefixed 20-byte hex address"`
  - `@Produce json`
  - `@Success 200 {object} wallet.TokenPortfolio`
  - `@Failure 400 {object} api.ErrorResponse`
  - `@Failure 502 {object} api.ErrorResponse`
  - `@Failure 503 {object} api.ErrorResponse`
  - `@Router /addresses/{address}/tokens [get]`
- `getTransactions`:
  - same `address` param, plus
  - `@Param limit query int false "page size (default 25, max 100)"`
  - `@Param pageKey query string false "opaque pagination cursor"`
  - `@Success 200 {object} wallet.TransactionPage`
  - same failure set and `@Router /addresses/{address}/transactions [get]`

swaggo reads the JSON tags on `wallet.Token`/`Transfer`/`Price`/`TokenPortfolio`/
`TransactionPage`, so response schemas (including the enriched `logoURI`/
`coinKey`/`priceUSD` fields) are documented from the Go types automatically.

### 2. Error response model

The handlers currently return `map[string]string{"error": msg}`, which swaggo
cannot describe as a schema. Introduce a small typed model and use it at the
write site so the error shape is documented:

```go
// ErrorResponse is the JSON body returned for any non-2xx response.
type ErrorResponse struct {
    Error string `json:"error"`
}
```

`writeError` is updated to write `ErrorResponse{Error: msg}` (same wire format —
`{"error": "..."}`), and the `@Failure` annotations reference
`{object} api.ErrorResponse`.

### 3. Generated spec package

`swag init --outputTypes json,yaml` outputs to a dedicated package
**`internal/apidocs/`** (not the existing `docs/` tree, which holds these
design specs):

```
internal/apidocs/
  swagger.json   # generated by swag
  swagger.yaml   # generated by swag
  embed.go       # hand-written; //go:embed swagger.json exposes SpecJSON []byte
```

`swag init` is run with `--outputTypes json,yaml`, which suppresses the
generated `docs.go` / `SwaggerInfo` entirely — swag never writes `docs.go`,
so it is never imported and never linked into the binary. The spec is served
solely via the `//go:embed` in the hand-written `embed.go`. These files are
committed so the app builds and the static page works without running `swag`.

### 4. App serving

Add to `internal/api/router.go` (root-level, outside `/v1`):

- `GET /openapi.json` → serves the spec JSON (`application/json`). Source is
  `apidocs.SpecJSON` — a `[]byte` exposed by the `internal/apidocs` package (see
  Decisions); the api package imports it and writes the bytes verbatim.
- `GET /docs` → serves a small static Redoc HTML page (Redoc via CDN) whose
  `<redoc spec-url="/openapi.json">` points at the spec route.

We do **not** use `swaggo/http-swagger` — it renders Swagger UI, whereas we want
Redoc's clean static style.

### 5. Static page (GitHub Pages / offline)

- `docs/api/index.html` — Redoc page (CDN) loading a sibling `openapi.json`.
- `docs/api/openapi.json` — a committed copy of the generated spec.

The doc-gen step copies the generated `internal/apidocs/swagger.json` to
`docs/api/openapi.json`, so the annotations remain the single source and the
static copy is never hand-edited. GitHub Pages can be pointed at `docs/`.

### 6. Regeneration workflow & freshness guard

- Track the `swag` CLI as a tool dependency via the Go 1.24+ `tool` directive
  in `go.mod` (`go get -tool github.com/swaggo/swag/cmd/swag`), invoked as
  `go tool swag`. No separate `tools.go` file with a build tag is needed.
- **`make docs`**: runs
  `go tool swag init -g cmd/server/main.go --output internal/apidocs --outputTypes json,yaml --parseDependency --parseInternal`,
  then copies `internal/apidocs/swagger.json` → `docs/api/openapi.json`.
  (`--parseDependency --parseInternal` so swaggo resolves the `wallet.*` response
  structs.)
- **`make docs-check`**: runs `swag init ...` (and the copy) then
  `git diff --exit-code` over `internal/apidocs/` and `docs/api/openapi.json`;
  fails if the committed spec is stale. This is the drift guard for the
  annotation-generated spec; wire it into CI.

## Data flow

```
handler annotations + main info
        │ swag init (make docs)
        ▼
internal/apidocs/ (swagger.json/yaml + embed.go, committed)
        │                         │ copy
        │ embedded                ▼
        ▼                   docs/api/openapi.json + index.html (committed, Pages)
app routes:
  GET /openapi.json  -> spec JSON
  GET /docs          -> Redoc HTML (loads /openapi.json)
```

## Error handling

- `/openapi.json` serves the embedded spec bytes; if the embed is somehow empty
  the route returns 500 (degenerate, caught by tests).
- `/docs` is a static HTML string; Redoc itself fetches `/openapi.json`
  client-side. If the CDN is unreachable the page won't render — acceptable for
  now; vendoring Redoc is the noted offline fallback.
- No new failure modes on the existing `/v1` endpoints.

## Testing

- **Route tests** (`internal/api/handlers_test.go` or a new `docs_test.go`):
  - `GET /openapi.json` → 200, `application/json`, valid JSON containing both
    expected paths and the `TokenPortfolio` schema.
  - `GET /docs` → 200, `text/html`, body references `/openapi.json`.
- **Static-copy sanity:** assert `docs/api/openapi.json` exists and parses as
  JSON (guards against an empty/broken committed copy).
- **Freshness:** `make docs-check` in CI (needs the `swag` binary, so it's a make
  target rather than a Go test).
- Existing handler/service tests remain unchanged and green.

## Decisions

- **Spec source for `/openapi.json`:** add a small committed `internal/apidocs/embed.go`
  with `//go:embed swagger.json` exposing `var SpecJSON []byte`. (`//go:embed`
  only sees files in its own package directory, so this lives in `apidocs`, not
  `api`.) The api package imports `apidocs` and serves `apidocs.SpecJSON`
  verbatim. Rejected alternative: reading swag's generated `SwaggerInfo` template
  via `swag.ReadDoc()` — it interpolates host/basePath and needs init-time
  registration, more moving parts than a raw embed.

  Note: `embed.go` is hand-written (not produced by `swag init`). Because `swag
  init` is run with `--outputTypes json,yaml`, it only writes `swagger.json` and
  `swagger.yaml`; it never touches `embed.go`, so `make docs` leaves it intact.

## Future options (out of scope)

- Upgrade to swag v2 for native OpenAPI 3.1 output.
- Vendor `redoc.standalone.js` for fully-offline `/docs` rendering.
- Add Scalar as an alternative renderer once on OpenAPI 3.x.
